    Diagram Guide¶

    Quick Reference for reading and navigating architecture diagrams in this documentation

    Overview¶

    This documentation uses Mermaid diagrams to visualize the lolstonks-api-gateway architecture. This guide explains how to read and interact with these diagrams effectively.

    Diagram Types Used¶

    Flowcharts¶

    Used for system architecture, data flows, and process workflows. Shows components and their relationships with directional arrows.

    Best for: Component architecture, request flows, decision trees

    Sequence Diagrams¶

    Used for temporal interactions between components. Shows the order of operations and communication patterns.

    Best for: Request/response cycles, API calls, error handling flows

    Graph Diagrams¶

    Used for network topology and multi-directional relationships. Shows complex interconnections between components.

    Best for: Provider registry, routing hierarchy, dependency graphs

    Reading Diagrams¶

    Top-to-Bottom Flow¶

    Most flowcharts follow a top-to-bottom or left-to-right reading pattern:

    1. Start: Entry points are typically at the top or left
    2. Follow arrows: Trace the path of execution
    3. Decision points: Diamond shapes indicate branches
    4. End points: Exit points are at bottom or right

    Subgraphs (Grouped Components)¶

    Diagrams use subgraphs to group related components:

    flowchart TD subgraph Gateway["Gateway Layer"] direction LR router[Router] middleware[Middleware] end subgraph Providers["Provider Layer"] direction LR riot[Riot API] dragon[Data Dragon] end Gateway --> Providers style Gateway fill:#e3f2fd,stroke:#1976d2,stroke-width:2px style Providers fill:#e8f5e8,stroke:#388e3c,stroke-width:2px

    Reading tip: Subgraphs represent architectural layers or functional groupings. Components within a subgraph are closely related.

    Performance Annotations¶

    Diagrams include real-world performance data:

    flowchart LR client[Client] -->|"HTTP Request<br/>(1-5ms network)"| gateway[API Gateway] gateway -->|"Cache Check<br/>(&lt;1ms)"| cache[(Redis Cache)] cache -->|"Cache Hit<br/>(&lt;10ms total)"| client cache -.->|"Cache Miss<br/>(50-200ms)"| riot[Riot API] riot -.->|"Cache Update<br/>(async)"| cache style gateway fill:#e3f2fd,stroke:#1976d2,stroke-width:2px style cache fill:#fff3e0,stroke:#f57c00,stroke-width:2px style riot fill:#ffebee,stroke:#c62828,stroke-width:2px

    Legend: - <Xms: Microseconds to low milliseconds (very fast) - X-Yms: Typical range in milliseconds - (async): Background operation, doesn't block response

    Common Patterns¶

    Request-Response Pattern¶

    sequenceDiagram participant C as Client participant G as Gateway participant P as Provider participant E as External API C->>G: HTTP Request G->>P: Get Data P->>E: API Call E-->>P: Response P-->>G: Processed Data G-->>C: HTTP Response Note over G,P: Caching occurs here

    Used in: All API endpoint flows, data retrieval operations

    Interactive Features¶

    Viewing Diagrams¶

    Diagrams in this documentation are rendered using Mermaid and support:

    1. Click to Zoom: Click on diagram elements for details (if supported by viewer)
    2. Pan: On complex diagrams, use pan gestures to navigate
    3. Export: Most viewers allow exporting to SVG/PNG

    Rendering Options¶

    GitHub/GitLab: Diagrams render automatically in markdown files

    VSCode: Install "Markdown Preview Mermaid Support" extension

    Documentation Sites: Use mermaid.js or similar renderer

    Live Editor: Use Mermaid Live Editor to edit and test diagrams

    Tips for Complex Diagrams¶

    1. Start with the Entry Point¶

    Look for components labeled "Client", "Request", or at the top-left of the diagram.

    2. Follow the Primary Path First¶

    Trace the main flow (usually the thickest or most direct arrows) before exploring branches.

    3. Understand Decision Points¶

    When you encounter a diamond shape, read the conditions on outgoing arrows to understand the branching logic.

    4. Group Related Components Mentally¶

    Notice subgraphs and color groupings—these represent logical layers or functional areas.

    5. Read Annotations¶

    Labels on arrows and notes provide critical context about timing, data types, or conditions.

    6. Reference the Legend¶

    If a component's purpose is unclear, check its color against the legend at the top of this guide.

    Diagram Conventions Summary¶

    Convention Meaning
    Top-to-bottom/Left-to-right Primary flow direction
    Color coding Component categorization (Gateway, Provider, Cache, External)
    Subgraphs Logical grouping of related components
    Solid arrows Synchronous calls
    Dotted arrows Async operations
    Thick arrows Primary data flow
    Diamond shapes Decision points
    Annotations Performance data, conditions, notes

    Getting Help¶

    If you're having trouble understanding a diagram:

    1. Check this guide for symbol meanings and conventions
    2. Read the surrounding text in the documentation
    3. Follow cross-references to related diagrams and docs
    4. Use the live editor to explore diagram structure interactively
